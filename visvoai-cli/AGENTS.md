# visvoai-cli

A terminal-native coding agent built on `visvoai-core` + `visvoai-ai`: a Textual
TUI (interactive REPL) plus a single-shot mode. The agent reads/edits files, runs
shell commands, searches the web, and renders its work ‚Äî streaming replies, tool
activity, diffs, thinking, and mermaid diagrams ‚Äî in the terminal.

Two surfaces, one console script (`visvoai`):
- `visvoai`            ‚Üí launch the Textual TUI
- `visvoai "prompt"`   ‚Üí single-shot: stream one turn to stdout

# Key Files
- `main.py` ‚Üí the `visvoai` entry point: no prompt ‚Üí `_launch_tui` (VisvoApp); a prompt ‚Üí `_run_single_shot` (CLIRuntime graph ‚Üí `astream_events` ‚Üí stdout). Both build the model via `visvoai.ai.build_chat_model`. Single-shot is headless: `--yes` auto-approves mutations; without it the gated tool set DENIES mutations except those `[permissions]` pre-authorizes (path confinement applies either way).
- `app.py` ‚Üí `VisvoApp` ‚Äî the Textual app: shell (CSS, bindings, compose/lifecycle, theme, welcome, quit/clear) + shared turn/HITL primitives (`_begin_turn`/`_mount_block`/`_tool_node`, `ask_choice`/`ask_form`/`ask_text`). Composed from mixins. Owns `_hitl_mode` + `action_cycle_hitl_mode` (shift+tab). Overrides `notify` to suppress info-level toasts (warning/error still show).
- `agent_turn.py` / `commands.py` / `sessions.py` / `render.py` / `demo.py` / `rewind.py` ‚Üí VisvoApp mixins (real turn, slash commands, resume/screens, answer+mermaid rendering, the test-only mock, git-structured history). Turn failures render via `ErrorBlock` (classified auth/network/model by `_classify_turn_error`), not a SystemNote. Persistence is **crash-durable**: the human message and each completed AI/Tool message are flushed via `_persist_turn` (idempotent tail-append) AS they stream, so a hard crash mid-turn loses nothing. The durable log is append-only and never trimmed; a dangling tool_call left by a crashed/errored turn is stripped at point-of-use by `_sanitize_thread` when building the model state (also covers resumed threads).
- `agent.py` ‚Üí pipeline glue: `build_agent_graph` (provider ‚Üí CLIRuntime graph; `enable_agents=True` adds the `run_agent` delegation tool), chunk classification, `SYSTEM_PROMPT`, deployment/usage/cost views. Textual-free.
- `agents.py` ‚Üí subagents: built-in roster (`explore` read-only ¬∑ `general` full) + user `.md` definitions (`~/.visvoai/agents/` ‚à™ `<project>/.visvoai/agents/`, project wins on name; frontmatter description/tools/model, body = system prompt). `build_run_agent_tool` ‚Üí `run_agent(agent, task)`: each dispatch builds a FRESH CLIRuntime graph (own prompt/tools, empty history; only the final answer returns); parallel dispatches run concurrently via ToolNode. Tool tiers fixed at graph build (`_tools_for_spec`): read-only ‚áí reads + write-refusing sandboxed shell (ungated), full ‚áí standard set + session MCP tools behind the same approve() gate (explicit lists may name `server__tool`; MCP never in read-only); neither includes run_agent (depth 1). Pending-trust agents are surfaced by the APP (startup + turn-end diff via `VisvoApp._notify_pending_agents`), never left to the model to mention. Subagent graphs run with the SAME context assembler as the main agent (env/cwd, project instructions, git) + a tool-reality-check prompt clause; their ainvoke is tagged `visvoai_subagent:<name>:<tool_call_id>` (id injected via InjectedToolCallId ‚Äî parallel same-name dispatches stay distinguishable) ‚Äî astream_events BUBBLES nested-graph events into the main turn stream, and the turn worker (agent_turn.py) + headless loop (main.py) filter on that tag or the subagent's private execution leaks into rendered/persisted main history. A run IS an agent conversation: structured Steps (agent_runs.py) rendered by the conversation's ToolRow widgets via `widgets/run_steps.py` (side panel tail + /runs full view); trace JSONL appended LIVE by the registry (meta‚Üísteps‚Üísummary; hung runs keep their partial transcript); the run_agent TOOL owns the lifecycle (register/cancel/finish ‚Äî it holds the dispatch id + cancellable task; the turn worker only feeds step_start/step_end paired by event run_id); /runs can stop one run (caller gets 'stopped by user', turn survives). Headless (no registry) writes a one-shot trace via `_write_headless_trace`. Project agents need first-use trust (`agent_trust.toml` ‚Äî hash of the WHOLE definition; the prompt is executable surface). `/agents` screen + `visvoai agents list/show/create/remove` (main.py).
- `turn_events.py` ‚Üí THE SORTER: routes sub-agent-tagged stream events into the run registry (+ returns their token spend); main events fall through to the conductor.
- `tool_render.py` ‚Üí THE PAINTER: finished tool call ‚Üí Style-B body (edit‚Üídiff, shell‚Üíexit, agent‚Üícard‚Ä¶); returns failed? for the conductor's nudges. New tool kinds add a branch here.
- `session_state.py` ‚Üí `SessionStateMixin`: cross-turn session truths (agent panel sync, pending-trust toasts) as shared helpers ‚Äî called from app timers, turn end, startup.
- `toolkit.py` ‚Üí the PUBLIC tool contract: `make_cli_tool(fn, gate='approve'|'self'|None, cap)` ‚Äî wraps a plain (a)sync fn into a house-rules tool (capped output, exceptions‚Üí'ERROR: ‚Ä¶' data, cleandoc'd description, schema from the author's signature) with the gate DECLARED as tool.metadata; `apply_gates()` is the one reader of that metadata. `load_user_tools()` imports ~/.visvoai/tools/*.py (GLOBAL ONLY ‚Äî importing is code execution; project layer gets TOML/MCP, never .py) collecting each module's `TOOLS` list; broken plugins log + skip.
- `specstore.py` ‚Üí `LayeredSpecStore` + `Layer` ‚Äî THE one implementation of the pattern mcp/agents/skills share: ordered-layer merge (later wins), coincident-layer identity dedupe (the $HOME/project_root bug class, killed generically), and `<kind>_trust.toml` spec-hash trust. Layers are providers (callables), not hardcoded dirs ‚Äî a DB-backed layer can implement the same shape later. Domain modules keep parsing + tool building only.
- `skills.py` ‚Üí skills: a skill = `~/.visvoai/skills/<name>/SKILL.md` ‚à™ `.visvoai/skills/<name>/SKILL.md` (project wins; flat `<name>.md` ok) ‚Äî frontmatter `description:` + optional `args:` block; BODY = instructions; supporting files next to SKILL.md read lazily via `read_skill(resource=‚Ä¶)` (progressive disclosure). Index + creation guidance live in the `read_skill` tool description (the CLI's run_agent-roster pattern, not a system-prompt provider). `$arg`/`$ARGUMENTS` substitution on the main body only. Skills = knowledge, never capability (mutations still gated) ‚Äî but project skills still need one-time trust (`skill_trust.toml`, whole-definition hash incl. resource NAMES). read_skill is in the main graph and EVERY subagent tier. `/skills` screen + `visvoai skills list/show/create/remove`. `[skills] extra_dirs` in either config layer aggregates EXTERNAL skill folders (e.g. ~/.claude/skills); trust tier follows the DECLARING config file (who controls the bytes), precedence global dir ‚Üí global extras ‚Üí project dir ‚Üí project extras.
- `runtime.py` ‚Üí `CLIRuntime(AgentRuntime)` ‚Äî agent‚Üítools loop, no interrupts/checkpointer; when built with a `ContextAssembler` it overrides `_build_agent_node` for per-turn prompt assembly (still honouring the soft-step-cap finalize), else falls back to the core static-prompt node.
- `context.py` ‚Üí `CLIContext(RuntimeContext)` ‚Äî cwd / terminal width; no auth/datastore.
- `store.py` ‚Üí folder-per-conversation persistence (no DB). Conversation: `meta.json` (title, `active_branch`, msg_count) + `checkpoints.jsonl` (the shared, append-only, immutable REGISTRY: checkpoint `id ‚Üí shadow commit`). Branches: `branches/<name>/` each with `thread.jsonl` + `receipts.jsonl` + `timeline.jsonl` (ordered `{checkpoint_id, message_index, kind, label}` ‚Äî that branch's turn/tool‚Üícheckpoint view) + `meta.json` (`tip`, `forked_from`). Conversation-level convenience (`save_conversation`/`append_messages`/`append_receipt`/`read_receipts`) delegates to the active branch. `project_root(cwd)` is the stable shadow-repo work-tree anchor. `find_project_id(cwd)` (read-only) + `has_conversations(pid)` (read-only) let the welcome banner detect launch state without writing the project anchor.
- `state.py` ‚Üí UI state, NOT in-repo. Per-project at `~/.visvoai/projects/<pid>/state.json` (`last_seen_version` for the "what's new" panel) + per-USER global at `~/.visvoai/state.json` (`used` features + `shown` one-time flags) driving adaptive tips & coachmarks. `record_used(feature)` / `used_features()`; `mark_shown(key)` (True the first time) / `was_shown(key)`. All reads are side-effect-free; writes are atomic + best-effort.
- `cli_changelog.py` ‚Üí parses the bundled `assets/CHANGELOG.md` via regex; exposes `parse_changelog()`, `new_since(last_seen)`, `one_line_summary(entry)`, `current_version()` (from installed metadata). Version compare is tuple-based so `'0.10.0' > '0.4.0'` is correct (string compare is not).
- `checkpoints.py` ‚Üí `ShadowRepo`: a separate GIT_DIR (`~/.visvoai/projects/<pid>/checkpoints.git`) with the project root as work tree ‚Äî snapshots the whole tree per tool-batch/turn (plumbing: add -A ‚Üí write-tree ‚Üí commit-tree, explicit parent, content-addressed dedup), TRUE-restores it, and supports refs / worktree fork / bundle. Never touches the user's own `.git`; honours `.gitignore` + default excludes (incl. `.visvoai` secrets). `available()` False ‚Üí callers skip checkpointing.
- `rewind.py` ‚Üí `RewindMixin`: records checkpoints during a turn (best-effort, never breaks a turn); `/rewind` (Ctrl+B) restore-or-branch; `/branch` switch/fork timelines; `/fork` worktree + seeded conversation; `/export` transcript or bundle; `/log` the chain. A branch reconstructs itself ONLY from its own folder (its `timeline.jsonl`), resolving a checkpoint id against the shared immutable registry ‚Äî never reading another branch's mutable state. Branch-from = deep-copy a branch folder + truncate (both timelines kept, fully isolated); `forked_from` is provenance only. See `cli-git-structure`.
- `keys.py` ‚Üí layered API-key resolution + storage: `load_keys_into_env(cwd)` (env > project secrets > global config), `set_key(provider, key, scope, cwd)` (0600, auto-gitignore for project).
- `catalog.py` ‚Üí `install_cli_catalog()` ‚Äî at startup installs `build_catalog([BakedSource(), RemoteModelsDevSource(cache)])` as the default registry so the picker shows the live models.dev catalog (cached `~/.visvoai/cache/models.json`, offline-tolerant). `--refresh-models` re-fetches. The picker (`agent.chat_deployments`) shows curated baked deployments always + a provider's full catalog only when keyed.
- `gated_tools.py` ‚Üí edit/write/shell behind a permission gate; edit/write also path-confined (`pathguard.confine`, boundary checked BEFORE the prompt); web tools ungated.
- `pathguard.py` ‚Üí `resolve_roots(cwd)` (cwd + `.visvoai/config.toml [permissions].write_roots`) + `confine(path, roots)` ‚Üí realpath-resolved, escape/symlink/`.git`-blocked; raises `PathDenied`. Confines WRITES only (reads free); `run_shell` is gate-governed, not confinable.
- `permissions.py` ‚Üí `PermissionPolicy.auto_allow(tool, args)` from `.visvoai/config.toml [permissions]` (`allow_shell` command prefixes, `allow_write` path globs); `load_policy(cwd)` layers global‚Üíproject. Pre-authorized ops skip the gate.
- `hitl_modes.py` ‚Üí `HITLMode` (NORMAL/AUTO_EDIT/ACCEPT_ALL): `next()` cycles, `auto_approves(tool)` (auto-edit = file writes only, shell stays gated), `chip` (None in NORMAL). Session-only; cycled by shift+tab (`action_cycle_hitl_mode`, priority binding) or `/mode`. Relaxes approval only ‚Äî path confinement is independent. Concurrent tool approvals are serialized by `app._hitl_lock` (ToolNode runs tool_calls in parallel) with a re-check inside the lock so a sibling's "allow all session" skips the redundant prompt.
- `processes.py` ‚Üí `ProcessRegistry`: app-level home for agent-started background processes (dev servers, watchers). Detached spawn (`start_new_session` ‚Üí group kill via killpg, SIGTERM‚ÜíSIGKILL), reader thread ‚Üí bounded ring buffer (2000 lines, per-reader cursors), `stop_all()` on app exit (`VisvoApp.on_unmount`) and headless `finally` ‚Äî no orphaned servers. Registry survives turn cancel, NOT app restart (tools report unknown ids as data).
- `tools/background.py` ‚Üí `build_background_tools(registry, cwd, approve)`: start_process / check_process(id, wait_seconds) / stop_process. Start/stop approval-gated, check free ‚Äî they self-gate, so build_agent_graph adds them UNwrapped (never through gate_tool). `stopped_by: user` surfaces to the agent on next check.
- `gitio.py` ‚Üí real git: working-tree status / `status_summary` (diff-free per-turn snapshot) / stage / unstage / commit / project files.
- `mcp.py` ‚Üí MCP servers: `[mcp_servers.<name>]` config (global `~/.visvoai/config.toml` ‚à™ project `.visvoai/config.toml`, project wins), `${VAR}` env expansion (secrets never in config), first-use trust for project-defined servers (hash of exec surface in `~/.visvoai/projects/<pid>/mcp_trust.toml` ‚Äî spec edit re-prompts, token rotation doesn't), session-cached discovery (`get_mcp_tools`, per-server 10s timeout, failures reported not raised; sessions are PERSISTENT ‚Äî one AsyncExitStack per project key in `_stacks`, tools bound to the live session so stateful servers (browsers) keep state across calls; `close_mcp_sessions()` on app exit, invalidate_cache() schedules close), and text-block TOML writers (`upsert_server_config`/`remove_server_config` ‚Äî preserve the rest of the file verbatim; validate-before-write). Transport is langchain-mcp-adapters/mcp SDK; tools exposed as `server__tool` LangChain tools, gated via `gate_tool` (copy, not mutation ‚Äî cached tools are re-gated every per-turn graph rebuild). `/mcp` lists status + walks trust approval; `visvoai mcp add/list/remove` manages config headlessly (main.py ‚Äî the entry point is a `_DefaultGroup` click group, `chat` is the default command so bare prompts still work; `if __name__` block must stay at EOF, below all command registrations).
- `context/` ‚Üí configurable per-turn system-prompt assembly (`build_assembler`); slots into `CLIRuntime._build_agent_node` (see `context/`).
- `mermaid.py` ‚Üí ```mermaid fence ‚Üí segments + a self-contained HTML viewer (pure helpers).
- `iconography.py` ‚Üí the ONE icon vocabulary (POINTER ‚ùØ, STATE_STYLE lifecycle map state‚Üí(icon, theme-token), GIT ‚éá, MODE_CHIP ‚óÜ reserved for the HITL chip only). Widgets import glyphs from here ‚Äî never improvise; colors come from the paired theme token via `theme.palette()`.
- `theme.py` / `grid.py` / `termbg.py` ‚Üí 12-palette theme (reads `palette_tokens.json`), grid alignment, terminal-bg detection.
- `tools/` ‚Üí the agent tool set (files/shell/web split; see `tools/`).
- `widgets/`, `screens/` ‚Üí reusable Textual widgets and full-screen views (each has its own AGENTS.md).
- `palette_tokens.json`, `assets/` ‚Üí shipped package data (see `[tool.setuptools.package-data]`).

# Key Classes / Functions
- `VisvoApp(App)` ‚Üí the Textual app; exposed lazily via `visvoai.cli.__getattr__` so `import visvoai.cli` / single-shot don't pull in Textual.
- `CLIRuntime(AgentRuntime)` ‚Üí no `_extend_graph` additions; optional `_build_agent_node` override for the context-assembly pipeline (`__init__(assembler=‚Ä¶)`).
- `cli` (main.py) ‚Üí Click command, the `visvoai` console script; routes prompt‚Üísingle-shot, no-prompt‚ÜíTUI.
- `build_agent_graph()` (agent.py) ‚Üí builds the graph the TUI streams; `build_chat_model` applies the deployment's default thinking.

# Conventions
- Pure public-package consumer: imports only `visvoai-ai`, `visvoai-core`, relative modules, and declared third-party deps (textual, rich, click, pygments, python-dotenv, langchain-google-genai). NEVER a private/platform module.
- No private names in shipped text (docstrings/comments/AGENTS.md) ‚Äî boundary rule 2.
- Every widget owns its styling via Textual `DEFAULT_CSS`; colours come only from `theme.py`.
- `app.py` holds the shell + shared primitives; feature concerns live in mixins.
- No graph interrupts/checkpointer ‚Äî the CLI is synchronous (the human is in the loop).

# Gotchas
- API keys: `main._bootstrap_env(cwd)` (both surfaces) loads `.env` then fills `os.environ` from the stored layers. Precedence is env > project (`.visvoai/secrets.toml`, gitignored) > global (`~/.visvoai/config.toml` `[api_keys]`); an exported var always wins. Keys NEVER go in the committed `.visvoai/config.toml`. Per-provider: the active model's provider determines which `{PROVIDER}_API_KEY` is needed; web search/fetch always need Gemini's. Add a key via `/login` (TUI) or `visvoai --set-key <provider>`.
- `astream_events` is called with an explicit `recursion_limit=100` (both surfaces) ‚Äî the LangGraph default 25 crashes a deep turn before the core graph's soft step cap can force a clean finalize.
- The mock showcase (DemoMixin + mock.py) is kept for tests but is NOT reachable from the menu, by typing `/demo`, or any key binding ‚Äî tests drive it via `run_command`.
- `theme.py` reads `palette_tokens.json` from its own directory at import time ‚Üí it must ship as package data (it does).
- Tests live in `tests/` and run in the package's own env (`uv run --extra dev pytest`); `[tool.uv.sources]` resolves the sibling packages by path for dev (pip/PyPI ignore that table).
- **Tool-result management** (`_prune_tool_results`, applied after `_sanitize_thread` when building model input): keeps recent tool results verbatim up to `TOOL_RESULT_BUDGET_CHARS`, elides older large ones to a stub (tool_call_id preserved). It transforms only the SENT copy ‚Äî the durable `_history` is never trimmed. When it elides (`n_elided>0`), the turn-end `self._history = list(final_messages)` adopt is **skipped** (final_messages was built from the pruned input; the incremental appends already hold full-fidelity new messages). Bounds cross-turn growth only; intra-turn tool-loop re-sends aren't touched (would need a per-call runtime hook or prompt caching).
- **Self-driving guidance** teaches by context and never nags: spinner tips are adaptive (`widgets.conversation.adaptive_tips(used)` drops a feature's tip once `state.record_used` has logged it); post-turn nudges + coachmarks self-silence (`_nudge_text` is pure/tested; coachmarks gated by `state.mark_shown`); `/tour` + `/help` + launch-state welcome + "what's new" are the discovery surfaces; unknown `/cmd` ‚Üí `commands._closest_command` (alias table then fuzzy). The proactive launch scan does real-git I/O so it's gated on `self.is_headless` (never runs under test). Keep the restraint: info-severity toasts are suppressed, panels self-silence ‚Äî that's what stops it feeling naggy.
- Git-structured history (rewind/branch/fork/export) is **best-effort**: a `CheckpointError` (or missing git) is always swallowed so it never breaks a turn ‚Äî `ShadowRepo.available()` gates it, and checkpoint tests `skipif` git is absent. TRUE restore is destructive on the active branch, but every checkpoint commit keeps a per-checkpoint ref (`refs/visvoai/<cid>/cp/<id>`) so discarded code stays recoverable; branching keeps both timelines outright and is **isolated** (deep-copied ‚Äî later changes to the parent never touch a fork). The shadow repo is per-project (shared, deduped, content-addressed commits); the checkpoint registry is shared and **immutable** (an id always maps to the same commit), which is what makes cross-branch id resolution safe. `/clear` resets the checkpoint tip so a fresh conversation re-baselines. Checkpointing REFUSES to run when the working dir is the home dir / filesystem root (`store.checkpoints_disabled_reason` ‚Äî anything containing `~/.visvoai`), and a snapshot failure disables it **loudly-once** (notify + `_cp_failed`) ‚Äî so `/rewind` being empty is never a silent mystery (the classic "launched visvoai in ~ ‚Üí git add -A on $HOME times out" trap).
