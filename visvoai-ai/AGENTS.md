# visvoai-ai

Unified multi-provider LLM access layer. Abstracts Gemini, Anthropic, OpenAI, and
every OpenAI-compatible endpoint behind a single `Provider` facade, plus a model
registry that is the single source of truth for model identity, pricing, and
capabilities.

# Key Files
- `src/visvoai/ai/__init__.py` ‚Üí public API surface (re-exports providers + registry helpers)
- `src/visvoai/ai/providers/base.py` ‚Üí `Provider` facade ABC + `NotSupported` + `default_content_events`
- `src/visvoai/ai/providers/gemini.py` ‚Üí `GeminiProvider`
- `src/visvoai/ai/providers/anthropic.py` ‚Üí `AnthropicProvider`
- `src/visvoai/ai/providers/openai_compat.py` ‚Üí `OpenAICompatProvider` + `ReasoningChatOpenAI`
- `src/visvoai/ai/providers/config.py` ‚Üí API-key + base_url resolution
- `src/visvoai/ai/providers/factory.py` ‚Üí `get_provider` / `get_provider_for_model` (name/model ‚Üí facade)
- `src/visvoai/ai/model_registry.py` ‚Üí baked model facts (ids, pricing, capabilities) + helpers ‚Äî the offline floor / default source
- `src/visvoai/ai/catalog/` ‚Üí catalog package: `engine.py` (`CatalogSource`/`BakedSource`/`build_catalog()` ‚Äî merge ‚Üí gate ‚Üí validate), `sources/` (one file per source: `modelsdev`, `remote`)
- `src/visvoai/ai/deployments.py` ‚Üí `DeploymentRegistry` (Model/Deployment view) + default-instance shell + `install_catalog()` seam

# Key Classes / Functions
- `Provider` ‚Üí base facade; `build_chat_model()` returns a streaming LangChain `BaseChatModel`, `normalize_content()` maps stream chunks to `{type, content}` events. Both optional (default `NotSupported`).
- `GeminiProvider` / `AnthropicProvider` / `OpenAICompatProvider` ‚Üí one subclass per family; lazily import their LangChain integration inside `build_chat_model()`.
- `resolve_api_key(provider, key)` / `resolve_base_url(provider, base_url)` ‚Üí explicit arg wins, else env var / built-in base_url default.
- `get_provider(name)` ‚Üí name ‚Üí facade (gemini/anthropic = bespoke; everything else = OpenAI-compat). `get_provider_for_model(id, capability=None)` ‚Üí registry-driven model id ‚Üí facade. Both **key-less** ‚Äî they pick the class; keys resolve at `build_chat_model()`.
- `CatalogSource` / `BakedSource` / `build_catalog(sources, gate=None)` ‚Üí stack sources (later wins, merged per `(provider, api_id)`), drop uncallable providers via `gate`, `validate`; returns `list[ModelDefinition]` ‚Äî drop-in for the static `MODELS` list.
- `DeploymentRegistry(model_defs)` ‚Üí instance-scoped Model/Deployment view (the correct core, no global state). Free funcs (`list_deployments`, `get_deployment`, `default_deployment`, ‚Ä¶) delegate to a module default instance; `install_catalog(defs)` / `set_default_registry(reg)` swap it once at startup.
- `catalog.sources.modelsdev.to_definitions(api_json)` / `ModelsDevSource` ‚Üí map a models.dev `api.json` ‚Üí `ModelDefinition`s. Admission is by **callability, not npm**: a provider is admitted if it has a derivable Chat Completions base_url (`api` field ‚Üí branded map) and isn't in `BESPOKE_OR_DENY`. ~4150 defs / ~128 providers from real data.
- `catalog.sources.remote.RemoteModelsDevSource(cache_path, url=‚Ä¶, ttl_seconds=‚Ä¶, fetcher=‚Ä¶, use_bundled_fallback=True)` ‚Üí cached, offline-tolerant models.dev fetch (stdlib `urllib` only). Degrades, never raises: fresh cache ‚Üí fetch ‚Üí stale cache ‚Üí **bundled snapshot** ‚Üí empty. Consumer owns the cache path; `fetcher=` is injectable for tests.
- `catalog.sources.remote.load_bundled_snapshot()` ‚Üí reads `catalog/data/modelsdev_snapshot.json.gz` (the OpenAI-compat offline floor, ~190 KB). Generated by `scripts/generate_modelsdev_snapshot.py` (deterministic, gzip mtime=0) ‚Äî committed lockfile, never hand-edited. Bespoke Gemini/Anthropic are NOT in it (they're the hand-curated `MODELS` floor).

# Conventions
- No dependency on any agent framework, datastore, web, or auth layer ‚Äî pure LLM access.
- Provider integrations import lazily ‚Äî the LangChain wrapper loads only when that provider's `build_chat_model()` is called.
- LangChain integration packages are declared as **per-provider extras** (`visvoai-ai[gemini]`, `[anthropic]`, `[openai]`, `[all]`) ‚Äî base install stays light.
- `AnthropicProvider.build_chat_model` reads `supports_thinking` from `model_registry` (single source of truth) ‚Äî never hardcode model-id checks in the provider.
- Public docstrings describe only the generic facade ‚Äî no private/consumer internals.

# Gotchas
- Together/Groq/OpenRouter/vLLM run on the **OpenAI-compat path** (`OpenAICompatProvider` ‚Üí `ChatOpenAI` + base_url), not a vendor SDK ‚Äî they ship under the `[openai]` extra; only an API key + base_url (auto-resolved in `config.py`) are needed.
- `OpenAICompatProvider.build_chat_model` raises `ValueError` if a non-`openai` provider has no resolvable base_url ‚Äî prevents silent fall-through to OpenAI's API with a foreign model id.
- `ReasoningChatOpenAI` is a thin `__new__` shim over a memoized `_reasoning_chat_openai_cls()` ‚Äî do not re-add per-call class creation.
- `OpenAICompatProvider.build` pins `use_responses_api=False` for non-`openai` providers: langchain-openai auto-switches to the OpenAI **Responses API** when a top-level `reasoning` dict is in the payload, but Together/OpenRouter/‚Ä¶ only speak Chat Completions (they 400 or return turn-breaking block-list content). Reasoning therefore rides `extra_body` ‚Äî see `thinking.OPENROUTER_REASONING`. Never send a top-level `reasoning` kwarg on a compat provider.
- Claude thinking is split by model version: `ThinkingMechanism.ANTHROPIC_ADAPTIVE` (4.6+: `{"type":"adaptive"}`) vs `ANTHROPIC_BUDGET` (‚â§4.5: legacy `budget_tokens`). `deployments._mechanism` routes by `api_id` prefix. Registry models still have `supports_thinking=False` (dialect is correct but live-unverified against the Anthropic API ‚Äî verify before flipping).
- `deployments.py` builds a default `DeploymentRegistry` from baked `MODELS` at import; a dynamic catalog is opted into by calling `install_catalog(build_catalog([...]))` once at startup. If no consumer calls it, behavior == baked-only (today). Integrity checks live in `DeploymentRegistry._check_integrity` (per-instance), not at import.
- `ModelDefinition.base_url` / `key_env` (and the same on `Deployment`) carry self-contained wiring for catalog-sourced providers NOT in the static config maps. `build_chat_model` resolves them to explicit `base_url=`/`api_key=` args so the **provider layer stays unchanged** (explicit wins). They're `None` for baked/static-wired entries; `resolve_api_key(provider, env_var=‚Ä¶)` accepts the carried key var.
