# visvoai-ai

Unified multi-provider LLM access layer. Abstracts Gemini, Anthropic, OpenAI, and
every OpenAI-compatible endpoint behind a single `Provider` facade, plus a model
registry that is the single source of truth for model identity, pricing, and
capabilities.

# Key Files
- `src/visvoai/ai/__init__.py` → public API surface (re-exports providers + registry helpers)
- `src/visvoai/ai/providers/base.py` → `Provider` facade ABC + `NotSupported` + `default_content_events`
- `src/visvoai/ai/providers/gemini.py` → `GeminiProvider`
- `src/visvoai/ai/providers/anthropic.py` → `AnthropicProvider`
- `src/visvoai/ai/providers/openai_compat.py` → `OpenAICompatProvider` + `ReasoningChatOpenAI`
- `src/visvoai/ai/providers/config.py` → API-key + base_url resolution
- `src/visvoai/ai/providers/factory.py` → `get_provider` / `get_provider_for_model` (name/model → facade)
- `src/visvoai/ai/model_registry.py` → baked model facts (ids, pricing, capabilities) + helpers — the offline floor / default source
- `src/visvoai/ai/catalog/` → catalog package: `engine.py` (`CatalogSource`/`BakedSource`/`build_catalog()` — merge → gate → validate), `sources/` (one file per source: `modelsdev`, `remote`)
- `src/visvoai/ai/deployments.py` → `DeploymentRegistry` (Model/Deployment view) + default-instance shell + `install_catalog()` seam

# Key Classes / Functions
- `Provider` → base facade; `build_chat_model()` returns a streaming LangChain `BaseChatModel`, `normalize_content()` maps stream chunks to `{type, content}` events. Both optional (default `NotSupported`).
- `GeminiProvider` / `AnthropicProvider` / `OpenAICompatProvider` → one subclass per family; lazily import their LangChain integration inside `build_chat_model()`.
- `resolve_api_key(provider, key)` / `resolve_base_url(provider, base_url)` → explicit arg wins, else env var / built-in base_url default.
- `get_provider(name)` → name → facade (gemini/anthropic = bespoke; everything else = OpenAI-compat). `get_provider_for_model(id, capability=None)` → registry-driven model id → facade. Both **key-less** — they pick the class; keys resolve at `build_chat_model()`.
- `CatalogSource` / `BakedSource` / `build_catalog(sources, gate=None)` → stack sources (later wins, merged per `(provider, api_id)`), drop uncallable providers via `gate`, `validate`; returns `list[ModelDefinition]` — drop-in for the static `MODELS` list.
- `DeploymentRegistry(model_defs)` → instance-scoped Model/Deployment view (the correct core, no global state). Free funcs (`list_deployments`, `get_deployment`, `default_deployment`, …) delegate to a module default instance; `install_catalog(defs)` / `set_default_registry(reg)` swap it once at startup.
- `catalog.sources.modelsdev.to_definitions(api_json)` / `ModelsDevSource` → map a models.dev `api.json` → `ModelDefinition`s. Admission is by **callability, not npm**: a provider is admitted if it has a derivable Chat Completions base_url (`api` field → branded map) and isn't in `BESPOKE_OR_DENY`. ~4150 defs / ~128 providers from real data.
- `catalog.sources.remote.RemoteModelsDevSource(cache_path, url=…, ttl_seconds=…, fetcher=…, use_bundled_fallback=True)` → cached, offline-tolerant models.dev fetch (stdlib `urllib` only). Degrades, never raises: fresh cache → fetch → stale cache → **bundled snapshot** → empty. Consumer owns the cache path; `fetcher=` is injectable for tests.
- `catalog.sources.remote.load_bundled_snapshot()` → reads `catalog/data/modelsdev_snapshot.json.gz` (the OpenAI-compat offline floor, ~190 KB). Generated by `scripts/generate_modelsdev_snapshot.py` (deterministic, gzip mtime=0) — committed lockfile, never hand-edited. Bespoke Gemini/Anthropic are NOT in it (they're the hand-curated `MODELS` floor).

# Conventions
- No dependency on any agent framework, datastore, web, or auth layer — pure LLM access.
- Provider integrations import lazily — the LangChain wrapper loads only when that provider's `build_chat_model()` is called.
- LangChain integration packages are declared as **per-provider extras** (`visvoai-ai[gemini]`, `[anthropic]`, `[openai]`, `[all]`) — base install stays light.
- `AnthropicProvider.build_chat_model` reads `supports_thinking` from `model_registry` (single source of truth) — never hardcode model-id checks in the provider.
- Public docstrings describe only the generic facade — no private/consumer internals.

# Gotchas
- Together/Groq/OpenRouter/vLLM run on the **OpenAI-compat path** (`OpenAICompatProvider` → `ChatOpenAI` + base_url), not a vendor SDK — they ship under the `[openai]` extra; only an API key + base_url (auto-resolved in `config.py`) are needed.
- `OpenAICompatProvider.build_chat_model` raises `ValueError` if a non-`openai` provider has no resolvable base_url — prevents silent fall-through to OpenAI's API with a foreign model id.
- `ReasoningChatOpenAI` is a thin `__new__` shim over a memoized `_reasoning_chat_openai_cls()` — do not re-add per-call class creation.
- Claude thinking is split by model version: `ThinkingMechanism.ANTHROPIC_ADAPTIVE` (4.6+: `{"type":"adaptive"}`) vs `ANTHROPIC_BUDGET` (≤4.5: legacy `budget_tokens`). `deployments._mechanism` routes by `api_id` prefix. Registry models still have `supports_thinking=False` (dialect is correct but live-unverified against the Anthropic API — verify before flipping).
- `deployments.py` builds a default `DeploymentRegistry` from baked `MODELS` at import; a dynamic catalog is opted into by calling `install_catalog(build_catalog([...]))` once at startup. If no consumer calls it, behavior == baked-only (today). Integrity checks live in `DeploymentRegistry._check_integrity` (per-instance), not at import.
- `ModelDefinition.base_url` / `key_env` (and the same on `Deployment`) carry self-contained wiring for catalog-sourced providers NOT in the static config maps. `build_chat_model` resolves them to explicit `base_url=`/`api_key=` args so the **provider layer stays unchanged** (explicit wins). They're `None` for baked/static-wired entries; `resolve_api_key(provider, env_var=…)` accepts the carried key var.
